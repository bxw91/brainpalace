# BrainPalace Benchmark Results

Baseline latency measurements for all five retrieval modes, generated from the
`docs/` corpus using `scripts/query_benchmark.py`.

---

## How to Run

```bash
# Prepare a fresh docs/ corpus and run a full baseline (recommended for reproducible comparisons)
python scripts/query_benchmark.py --prepare-docs-corpus

# JSON output for CI or scripted comparisons
python scripts/query_benchmark.py --json

# Quick table view against existing index (no corpus reset)
python scripts/query_benchmark.py
```

The query set is loaded from `scripts/benchmark_queries.json` (20 fixed queries, version-controlled).
Results are comparable across runs only when the same corpus and query set are used.

## Notes

### Measurement Methodology

- **Client latency** is measured with `time.perf_counter()` around `httpx.post("/query/")`.
  It includes HTTP round-trip, serialization, and all server-side processing.
- **Server latency** (`query_time_ms`) is the field returned in the API response,
  measuring time inside the query handler only.
- **Warm-up queries** (3 per mode, discarded) allow ChromaDB, BM25, and embedding
  caches to warm before timed measurements.
- **Corpus:** Fresh `docs/` index with 482 chunks (Markdown documentation files only,
  no code). Prepared with `--prepare-docs-corpus` which resets the index and re-indexes
  before timing.
- **Serial execution:** Queries are sent one at a time (no concurrency). QPS reflects
  single-client throughput, not server saturation capacity.

### Why graph mode is fast on Chroma

When GraphRAG is enabled with `store_type: simple`, the graph store is in-memory.
Graph traversal is a dictionary lookup, not a network round-trip, so server-side
`query_time_ms` for graph mode (~2.3 ms) is comparable to vector mode (~2.3 ms).
Multi mode is slower (~4.8 ms server-side) because it combines results from multiple
retrieval paths.

### Reproducing These Numbers

To reproduce results comparable to this baseline:

1. Ensure you are using the `docs/` corpus: `python scripts/query_benchmark.py --prepare-docs-corpus`
2. Use the default query set: `scripts/benchmark_queries.json` (do not modify the file)
3. Use the same backend and GraphRAG configuration shown in the metadata table
4. Machine-level differences (CPU speed, disk I/O, OS scheduler) will cause variation

## Interpreting Results

| Metric | Meaning |
|--------|---------|
| **p50** | Median latency — half of queries complete faster than this |
| **p95** | 95th percentile — 1 in 20 queries takes longer than this |
| **p99** | 99th percentile — 1 in 100 queries takes longer than this |
| **Mean** | Arithmetic mean — can be skewed by outliers; use p50 for typical performance |
| **QPS** | Queries per second — computed as `count * 1000 / total_ms` (single client, serial) |
| **ok** | Mode ran successfully; latency values are valid |
| **unsupported** | Mode not available on current backend/graph config; reason shown in Status |
| **skipped** | Mode excluded from this run via `--modes` flag |
